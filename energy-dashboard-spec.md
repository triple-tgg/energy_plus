# เอกสารออกแบบ — Energy Monitoring Dashboard (ต้นแบบ)

ระบบ Dashboard แสดงผลการใช้พลังงานแบบ drill-down หลายระดับ + ตาราง Realtime + กราฟเปรียบเทียบ
สร้างจากแบบร่าง (รูปที่ Design ไว้) เป็นต้นแบบที่คลิกเล่นได้จริง โดยใช้ข้อมูลจำลองที่อิงตาม schema `meter_data_realtime`

- ไฟล์ต้นแบบ: `energy-dashboard.jsx` (React artifact)
- เทคโนโลยี: React, recharts (กราฟ), lucide-react (ไอคอน), Tailwind core + inline style
- สถานะงาน: เฟส 1–3 เสร็จครบ

---

## 1. ภาพรวมระบบ

Dashboard มี 2 โหมดหลักสลับบนแถบด้านบน

| โหมด | หน้าที่ |
|------|---------|
| **ติดตาม Realtime** | ไล่ดูสถานะ/ค่าพลังงานแต่ละจุดแบบเรียลไทม์ ตั้งแต่ระดับสาขาลงไปถึงมิเตอร์ |
| **เปรียบเทียบ** | กราฟแท่ง/วงกลมเปรียบเทียบการใช้พลังงานข้ามหน่วยและช่วงเวลา |

เลือกชนิด Sensor ได้ที่แถบบน (ปัจจุบันเปิดใช้ **ไฟฟ้า**; น้ำ/แก๊ส/Solar เป็น placeholder)

---

## 2. โครงสร้างการนำทาง (Hierarchy)

```
Sensor (ไฟฟ้า/น้ำ/แก๊ส/Solar)
   └─ สาขา (Branch)
        └─ ตึก (Building)
             └─ ชั้น (Floor)
                  └─ โซน (Zone)
                       └─ ห้อง (Room) = Meter / Sensor ID   (1 Loop ≤ 32 ตัว)
```

มี breadcrumb กดย้อนกลับได้ทุกระดับ และตาราง Realtime ของระดับปัจจุบันอยู่ด้านขวาเสมอ

---

## 3. มุมมองตามระดับ (Phase 1–3)

| ระดับ / โหมด | มุมมอง | อ้างอิงแบบร่าง |
|---|---|---|
| สาขา, ตึก | การ์ดเรียงจากใช้มากไปน้อย + แถบแจ้งเตือน | รูป ①, รูป 2 |
| ชั้น | **ผังด้านข้างอาคาร** (ชั้นสูงอยู่บน → ชั้น 1 อยู่ล่าง) สลับเป็น **Single Line Diagram** ได้ | รูป 2.1, 2.4 |
| โซน | **ผังพื้นที่** (โซน A/B/C/D เป็นช่อง) | รูป 2.2 |
| ห้อง | **กล่องมิเตอร์จัดตาม Loop** + ป้ายโหมด Auto/กรอกมือ/ปิดใช้งาน | รูป 2.3 |
| ทุกระดับ | ตาราง Realtime ด้านขวา (No · ชื่อ/รหัส · kWh · STS · อัปเดต) เรียงได้ | รูป ① |
| คลิกมิเตอร์ | หน้า detail: ค่า 3 เฟส (V/A/kW/PF/Hz/kVA/kVAR) + `import_kwhr` + พลังงานงวดนี้ + metadata | — |
| โหมดเปรียบเทียบ | กราฟแท่ง stacked + วงกลม % เลือกมิติ/ช่วงเวลาได้ | รูป 3, 3.1, 3.2 |

## 4. ระบบสถานะสี (STS) และโหมดข้อมูล

### สถานะสี
| สี | ความหมาย | เงื่อนไข (ในต้นแบบ — ปรับได้) |
|---|---|---|
| 🟢 เขียว | ปกติ | พลังงานงวดนี้ ≤ 80% ของเกณฑ์ |
| 🟡 เหลือง | เตือน | 80–100% ของเกณฑ์ |
| 🔴 แดง | เกินเกณฑ์ | > 100% ของเกณฑ์ |
| ⚪ เทา | ไม่มีสัญญาณ / ปิดใช้งาน | `received_at` ค้างเกิน 30 วินาที หรือถูกปิดใช้งาน |

> สถานะของระดับบน (โซน/ชั้น/ตึก/สาขา) สรุปจากลูกแบบ worst-case: แดง > เหลือง > เขียว > เทา

### โหมดป้อนข้อมูล (input mode)
| โหมด | สัญลักษณ์ | หมายเหตุ |
|---|---|---|
| Auto | ⚡ | อ่านค่าอัตโนมัติจากอุปกรณ์ (ปกติ) |
| กรอกมือ (Manual) | ✎ เหลือง | ป้อนค่าด้วยมือ |
| ปิดใช้งาน (Disabled) | ⏻ เทา | ไม่นับ/ไม่ส่งสัญญาณ |

## 5. ตรรกะช่วงเวลาของกราฟ (อ้างอิงรูป 3.3)

| รูปแบบ | แกน X | หน้าต่างข้อมูล |
|---|---|---|
| รายปี | 12 เดือน | ค่ามิเตอร์ปลายปี − ต้นปี (เช่น 00:00 1 ม.ค. 2569 − 00:00 1 ม.ค. 2568 = ทั้งปี 2568) |
| รายเดือน | วัน 1–30/31 | **รอบปฏิทิน** (1–สิ้นเดือน) หรือ **รอบบิล** (ตัดวันที่ 20 → 20 เดือนถัดไป) |
| รายสัปดาห์ | จ.–อา. | 7 วันของสัปดาห์ |
| รายวัน | 24 ชั่วโมง | 00:00–24:00 ของวัน |

โหมดเปรียบเทียบมีแถบแสดงหน้าต่างเวลาที่ใช้จริง และสลับรอบปฏิทิน/รอบบิลได้เมื่อเลือก "รายเดือน"

## 6. โมเดลข้อมูล — การ map `meter_data_realtime`

| ฟิลด์ใน schema | ใช้ใน Dashboard |
|---|---|
| `site_id` | สาขา (Branch) |
| `address_id` | ตำแหน่งติดตั้ง → ตึก/ชั้น/โซน/ห้อง **(ดูข้อ 9 ข้อ 1)** |
| `channel`, `device`, `code`, `type` | ระบุตัว Meter / Sensor ID |
| `vl1–vl3`, `vl12/23/31` | แรงดัน L-N / L-L (หน้า detail) |
| `il1–il3` | กระแสแต่ละเฟส |
| `kw1–kw3`, `kw_3ph` | กำลังไฟ ณ ขณะนั้น (kW) |
| `kvar*`, `kva*`, `pf1–pf3`, `hz` | ค่าคุณภาพไฟ (หน้า detail) |
| `import_kwhr` | เลขมิเตอร์สะสม → **คำนวณพลังงานต่อช่วง** |
| `device_datetime`, `received_at` | เช็คความสด → ถ้าค้าง = เทา (ไม่มีสัญญาณ) |
| `raw_json` | payload ดิบ |

## 7. สถาปัตยกรรมข้อมูล

```
อุปกรณ์ IoT (Meter)
   ├─ Redis (ค่าล่าสุด)        → มุมมอง Realtime (ตาราง · การ์ด · สถานะสี · detail)
   └─ PostgreSQL              → โหมดเปรียบเทียบ (กราฟย้อนหลัง)
       meter_data_realtime
```

- **Realtime:** อ่านค่าล่าสุดจาก Redis
- **ย้อนหลัง:** คิวรีจาก PostgreSQL โดยพลังงานต่อช่วง = `import_kwhr(ปลายงวด) − import_kwhr(ต้นงวด)` ตามหน้าต่างเวลาที่เลือก

---

## 8. สูตรการคำนวณ (ในต้นแบบ)

- **พลังงานงวดนี้ (period kWh)** = `import_kwhr − periodStart_kwhr`
- **ค่ารวมของกลุ่ม** = ผลรวมพลังงานงวดของมิเตอร์ลูกทั้งหมด
- **สถานะกลุ่ม** = worst-case ของลูก
- **สัดส่วน % (วงกลม)** = ค่าหน่วยนั้น ÷ ผลรวมทุกหน่วยในช่วงที่เลือก
- ค่าพารามิเตอร์จำลอง: รอบรีเฟรช 2 วินาที, เกณฑ์ค้างสัญญาณ 30 วินาที (ของจริงตั้งตามรอบส่งข้อมูลอุปกรณ์)

---

## 9. สมมติฐาน & รายการที่ยังต้องยืนยัน

1. **ตาราง map ลำดับชั้น (สำคัญสุด)** — schema มีแค่ `site_id` + `address_id` ยังต้องการตารางที่ map `address_id` → ตึก/ชั้น/โซน/ห้อง/loop เพื่อแทนโครงสร้างจำลอง
2. **โครงสร้าง Redis** — รูปแบบ key/value ของค่าล่าสุด
3. **เกณฑ์สีสถานะ** — แหล่งที่มาของ threshold (% เทียบ baseline หรือค่าคงที่ต่อมิเตอร์) และเกณฑ์ค้างสัญญาณจริง
4. **หน้าตัวอย่างตาราง Realtime** — ที่อ้างถึงในรูป ①
5. **หน่วยของ Sensor อื่น** — น้ำ (m³?), แก๊ส, Solar เพื่อรองรับการสลับหน่วย

---

## 10. การนำไปต่อข้อมูลจริง

จุดที่ออกแบบให้สลับจาก mock → ข้อมูลจริงได้ง่าย:

- แทนฟังก์ชัน `generateData()` ด้วยการดึงโครงสร้างจากตาราง mapping (ข้อ 9.1)
- มุมมอง Realtime: subscribe/poll ค่าจาก Redis แทน loop จำลองทุก 2 วินาที
- โหมดเปรียบเทียบ: เปลี่ยน `data` (entity × bucket) ให้มาจากคิวรี PostgreSQL ตามหน้าต่างเวลา — โครง entity/bucket วางไว้รองรับแล้ว
- เกณฑ์สี/เกณฑ์ค้างสัญญาณ: ปรับที่ค่าคงที่ `STATUS`, `STALE_MS` และ `threshold` ต่อมิเตอร์

---

## 11. แผนถัดไป (เสนอ)

- คลิกแท่ง/ชิ้นพายในกราฟเพื่อเจาะลึก (drill จากกราฟ)
- ปรับ responsive สำหรับจอเล็ก/มือถือ
- หน้า Sensor อื่น (น้ำ/แก๊ส/Solar) พร้อมหน่วยเฉพาะ
- เชื่อม Redis + PostgreSQL จริง และระบบแจ้งเตือน/ประวัติเหตุการณ์
